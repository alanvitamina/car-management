# 公务用车管理系统：系统架构与流程边界 V1

## 1. 文档目的

本文档用于明确公务用车管理系统的建设目标、总体架构、系统边界、业务流程边界与集成策略，作为后续产品设计、技术设计、项目排期与开发实施的基础输入。

本版本重点回答以下问题：

- 为什么需要新建公务用车管理 Web 系统
- 新系统与飞书、OA 的职责边界如何划分
- 审批流程应放在何处承载
- Agent 在该系统中的定位与作用是什么
- 后续标准软件开发应如何分阶段推进

## 2. 项目背景与建设目标

### 2.1 当前问题

当前公司公务用车相关流程主要依赖 OA 系统承载审批，配合线下沟通、人工派车、人工核算完成闭环，存在以下问题：

- 用车申请、审批、派车、费用核算分散在多个系统和线下动作中，流程割裂
- 员工无法在申请时实时查看车辆可用状态，体验较差
- 行政经理需要人工判断车辆与司机是否冲突，派车效率低
- 私车公用补助、司机补助等依赖人工核算，周期长且容易出错
- 用车数据缺乏统一沉淀，难以支撑后续分析和智能化能力
- OA 与飞书并存，审批触达与使用体验不统一

### 2.2 建设目标

本项目不是单纯建设一个“审批表单系统”，而是建设一套面向 AI 驱动业务重构的公务用车业务底座。

建设目标如下：

- 建立覆盖“申请-审批-派车-执行-确认-核算-分析”的完整业务闭环
- 通过统一 Web 系统沉淀结构化业务数据和标准化业务状态
- 通过飞书提供统一入口、登录、消息通知与审批触达能力
- 尽量减少对 OA 主流程的依赖，逐步将适合迁移的通用审批场景迁出 OA
- 为后续 Agent 落地提供可执行、可追踪、可解释的业务基础

## 3. 建设原则

### 3.1 业务主系统唯一

公务用车业务的核心数据、业务规则、状态流转应由新建 Web 系统统一承载，避免多个系统分别维护状态造成不一致。

### 3.2 审批与业务解耦

审批是业务流程中的一个环节，但不是全部。审批承载可适度借助飞书能力，但业务规则、冲突校验、派车、核算、状态控制必须掌握在业务系统内。

### 3.3 统一入口优先

员工使用体验优先，尽量通过飞书提供统一访问入口、消息提醒和待办触达，降低“多系统割裂感”。

### 3.4 OA 集成可选、渐进迁移

对 OA 保持开放态度，但不默认 OA 为主业务引擎。是否接入 OA，应根据公司制度要求、历史流程依赖和场景敏感性分级判断。

### 3.5 Agent 以“提效闭环”为目标

Agent 不应脱离业务系统单独存在，而应建立在统一业务底座上，承担推荐、校验、催办、解释、自动执行等高价值工作。

## 4. 总体架构

### 4.1 架构定位

建议采用“三层协同”架构：

- 业务核心层：公务用车管理 Web 系统
- 协同入口层：飞书
- 存量系统层：OA

### 4.2 各层职责

#### 4.2.1 公务用车管理 Web 系统

作为业务主系统，负责：

- 车辆档案管理
- 司机档案管理
- 用车申请管理
- 审批状态管理
- 车辆与司机冲突校验
- 派车与资源锁定
- 用车消耗记录与确认
- 私车公用补助核算
- 司机补助核算
- 数据统计分析
- Agent 执行动作与决策支持

#### 4.2.2 飞书

作为统一协同入口，负责：

- 统一访问入口
- 单点登录或身份映射
- 消息通知、待办提醒
- 审批触达与审批动作承载
- 面向员工与管理者的轻量交互入口
- 后续 Agent 对话入口

#### 4.2.3 OA

作为存量系统，可选承担：

- 历史流程保留
- 暂时无法迁移的审批场景承载
- 制度要求必须留存在 OA 的流程归档
- 与组织架构、人员信息等既有数据的辅助同步

不建议 OA 继续承载本系统的核心业务规则与主状态流转。

## 5. 系统边界定义

### 5.1 新系统负责的内容

以下内容必须在新系统中实现：

- 申请表单与业务单据管理
- 车辆实时可用状态展示
- 车辆时间冲突校验
- 司机时间冲突校验
- 审批前业务合法性校验
- 审批通过后的车辆与司机分配
- 车辆状态变更与恢复
- 申请取消、修改、重新发起
- 出车消耗录入
- 消耗确认
- 补助自动核算
- 全量报表和数据看板
- Agent 规则执行、数据读取与操作入口

### 5.2 飞书负责的内容

以下内容建议由飞书承载：

- 应用入口
- 用户登录与身份映射
- 审批消息触达
- 审批动作入口
- 催办通知
- 任务提醒
- Agent 交互入口

### 5.3 OA 负责的内容

以下内容仅在确有必要时保留在 OA：

- 暂时无法迁移的旧流程
- 公司要求必须保留在 OA 的审批场景
- 需与既有 OA 流程体系并行过渡的阶段性流程

### 5.4 不建议的边界设计

以下做法不建议采用：

- 将业务主状态维护在 OA，业务系统仅做展示
- 将申请主表单直接完全建设在飞书审批表单中
- 在多个系统中同时维护车辆状态、审批状态、派车状态
- 让 Agent 直接基于聊天消息做核心业务判断而不依赖结构化数据

## 6. 审批承载方案评估

## 6.1 方案 A：审批与业务全在新系统内

### 说明

申请、审批、派车、确认、核算全部由新系统负责，飞书仅做登录和消息通知。

### 优点

- 业务闭环最完整
- 状态一致性最好
- 可定制能力最强

### 缺点

- 需要自行建设审批流能力
- 员工审批入口与现有飞书习惯不完全一致

### 结论

适合对审批引擎有较强自主要求的场景，但不作为当前优先推荐方案。

## 6.2 方案 B：新系统做业务主控，飞书做审批承载

### 说明

申请由新系统创建业务单据并完成实时校验，审批任务同步到飞书，审批结果回传新系统，由新系统继续执行派车、锁车、通知、确认、核算等动作。

### 优点

- 员工体验更统一
- 无需自建复杂 BPM 引擎
- 业务规则仍由新系统控制
- 适合后续 Agent 能力接入

### 缺点

- 需要做好飞书审批与业务状态的映射
- 需处理审批撤回、取消、改期后的回写逻辑

### 结论

这是当前最推荐的方案。

## 6.3 方案 C：OA 做审批主系统，新系统做业务辅助

### 说明

申请由新系统发起后，同步到 OA 审批，审批通过后再回写新系统。

### 优点

- 保持与历史制度兼容
- 方便满足必须留痕在 OA 的要求

### 缺点

- 流程割裂问题仍然明显
- 修改、取消、撤回、重提等逻辑复杂
- 审批状态、资源锁定与业务动作容易不同步
- 后续 Agent 价值会被系统边界削弱

### 结论

可作为过渡或个别高要求场景的备选，不建议作为本项目默认架构。

## 7. 推荐总体方案

推荐采用方案 B：

- 新系统作为业务主系统和唯一业务状态源
- 飞书作为统一入口、审批承载和消息通知平台
- OA 仅作为可选保留系统，不承担主业务链路

推荐后的主边界如下：

- 申请填写：新系统
- 审批动作：飞书
- 业务状态控制：新系统
- 派车与冲突校验：新系统
- 通知提醒：飞书
- 数据看板：新系统
- Agent 入口：飞书
- Agent 规则执行：新系统

## 8. 业务流程边界

### 8.1 流程总览

完整业务流程建议定义为：

1. 员工在飞书进入公务用车系统
2. 员工在新系统中发起用车申请
3. 新系统实时校验申请合法性
4. 新系统生成业务申请单
5. 新系统将审批任务推送到飞书
6. 一级审批人在飞书完成审批
7. 二级审批人在飞书完成审批
8. 审批结果回写新系统
9. 新系统完成车辆与司机分配、资源锁定与状态更新
10. 司机执行任务并录入消耗
11. 用车人确认消耗
12. 新系统完成补助核算、状态恢复、数据沉淀
13. Agent 基于全流程数据完成辅助决策与自动化动作

### 8.2 各节点边界

#### 节点 1：申请发起

- 承载系统：新系统
- 触达入口：飞书
- 核心动作：
  - 选择用车类型
  - 选择开始时间、结束时间、目的地、事由
  - 展示可用车辆状态
  - 判断是否存在时间冲突
  - 自动带出一级、二级审批人

#### 节点 2：审批流转

- 承载系统：飞书
- 主状态源：新系统
- 核心动作：
  - 新系统推送审批任务
  - 审批人在飞书处理通过或驳回
  - 审批结果回写新系统

#### 节点 3：派车与资源锁定

- 承载系统：新系统
- 核心动作：
  - 校验车辆状态是否仍可用
  - 校验司机状态是否仍可用
  - 分配车辆和司机
  - 将车辆状态更新为“已派出”或“已预占”
  - 向申请人和司机发送飞书通知

#### 节点 4：执行与消耗记录

- 承载系统：新系统
- 核心动作：
  - 司机录入出车前后里程
  - 司机录入油费、路桥费、停车费等
  - 上传必要凭证

#### 节点 5：消耗确认

- 承载系统：新系统
- 触达方式：飞书通知
- 核心动作：
  - 用车人确认消耗记录
  - 记录锁定
  - 作为核算依据

#### 节点 6：核算与结案

- 承载系统：新系统
- 核心动作：
  - 计算私车公用补助
  - 计算司机补助
  - 恢复车辆状态
  - 进入已完成状态

## 9. 关键状态设计

### 9.1 申请单状态

建议至少包含以下状态：

- 草稿
- 待一级审批
- 一级已通过
- 待二级审批
- 已驳回
- 审批通过待派车
- 已派车待执行
- 执行中
- 待消耗确认
- 已完成
- 已取消

### 9.2 车辆状态

建议至少包含以下状态：

- 空闲
- 已预占
- 已派出
- 维修中
- 停用

说明：

- 若审批通过后与实际出车存在时间差，建议引入“已预占”状态，以支持排期和冲突控制。

### 9.3 司机状态

建议至少包含以下状态：

- 在岗可派
- 已预占
- 出车中
- 休息
- 请假

## 10. 取消、修改、重新发起的边界处理

### 10.1 取消申请

- 发起人可在审批前或审批中发起取消
- 若审批在飞书中，需同步结束或作废对应审批实例
- 若已派车，取消需释放车辆和司机资源

### 10.2 修改申请

- 已提交申请原则上不直接覆盖原单据
- 建议采用“变更申请”或“撤回后重提”机制
- 若修改涉及时间、地点、用车类型、资源分配，需重新校验和重新审批

### 10.3 重新发起

- 支持基于原申请复制生成新单据
- 保留原单据和新单据的关联关系，便于追溯

## 11. Agent 在系统中的定位

## 11.1 Agent 不是审批引擎替代者

Agent 的核心价值不是代替表单系统，而是在标准业务流程之上减少人工判断、催办、核算和查询工作量。

## 11.2 第一阶段建议落地的 Agent 能力

- 智能派车建议
- 审批催办 Agent
- 司机/车辆冲突预警
- 费用与里程异常识别
- 私车公用补助自动解释
- 行政经理自然语言查询助手

## 11.3 Agent 的依赖前提

以下前提未完成前，不建议直接大规模投入 Agent 开发：

- 核心业务状态清晰
- 关键数据结构化
- 关键动作有接口可调用
- 审批、派车、确认、核算的责任边界明确

## 12. OA 集成策略建议

### 12.1 总体建议

对 OA 采取“开放但克制”的策略。

### 12.2 建议保留 OA 的情况

- 流程具有制度刚性，必须留痕在 OA
- 涉密或高敏场景暂不宜迁出
- 某些历史系统强依赖 OA 流程编号或归档机制

### 12.3 建议迁出 OA 的情况

- 通用行政审批
- 不涉密、规则明确、需要高频操作的场景
- 强依赖实时状态与自动计算的流程
- 需要后续接入 Agent 的流程

### 12.4 过渡期建议

建议按场景分级推进，而不是一次性全面替换：

- 第一阶段：公务用车流程独立建设，默认走飞书审批
- 第二阶段：视组织接受度决定是否将更多行政类流程迁出 OA
- 第三阶段：形成统一的“业务系统 + 飞书入口 + Agent 增强”的模式

## 14. 当前推荐结论

当前阶段建议形成以下共识：

- 本项目应建设为独立 Web 业务系统
- 飞书作为统一入口和审批承载平台优先接入
- OA 不作为默认主流程系统，仅保留必要集成空间
- 新系统必须掌握业务状态、规则、资源分配和核算能力
- Agent 建设应基于该系统逐步增强，而不是脱离系统单独建设
