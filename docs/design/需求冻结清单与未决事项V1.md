# 公务用车管理系统：需求冻结清单与未决事项 V1

## 1. 文档说明

本文档用于在项目正式开工前，冻结第一期建设范围、关键业务口径、交付边界与仍需管理的未决事项，作为产品、开发、测试、实施共同遵循的基线文档。

本文档的目标不是替代 PRD、需求分析、页面原型和接口文档，而是回答两个问题：

- 第一期开工时到底按什么范围做
- 仍未完全确认的事项，默认按什么口径推进

## 2. 冻结原则

### 2.1 范围冻结原则

- 第一期开工以 `P0` 范围为准，`P1/P2` 不得混入首批开发承诺
- 本文档未明确纳入第一期的能力，一律视为不在 `P0`
- 若后续新增需求影响数据库主模型、主流程状态机或飞书审批模板，必须走变更评审

### 2.2 口径冻结原则

- 业务口径冲突时，以本文档和 [业务规则清单V1.md](业务规则清单V1.md>) 为准
- 页面交互冲突时，以 [页面清单与页面原型定稿V2.md](页面清单与页面原型定稿V2.md>) 为准
- 飞书、OA 等外部系统集成口径冲突时，以 [飞书与OA集成设计说明V2.md](飞书与OA集成设计说明V2.md>) 为准

### 2.3 未决事项推进原则

- 仍需进一步确认的事项，不阻塞第一期脚手架、数据库、主流程和大部分页面开发
- 未决事项若在对应开发节点前未被业务方书面推翻，则默认采用本文档中的“冻结建议”

## 3. 第一期开工冻结范围

### 3.1 第一阶段必须交付的 P0 能力

第一期正式开工默认纳入以下 `P0` 范围：

- 飞书入口与登录态识别
- 组织架构同步与审批人自动带出
- 车辆档案管理
- 司机档案管理
- 公务用车申请
- 私车公用申请
- 飞书审批创建、回调、状态回写
- 申请取消
- 申请变更与重新发起基础版
- 待派车列表
- 车辆与司机可用性查询
- 派车与资源预占
- 公务用车消耗录入
- 公务用车消耗确认与驳回
- 私车公用里程与费用录入
- 私车公用导航核验与核算里程确认
- 私车公用补助核算
- 司机补助核算
- 基础通知提醒
- 基础工作台和基础看板
- 审计日志与集成日志

### 3.2 明确不纳入第一期承诺的能力

以下能力冻结为不在第一期 `P0` 承诺内：

- 财务系统自动结算对接
- OA 全量流程并行迁移
- 司机端离线能力
- 复杂排班引擎
- 多车队、多法人、多城市运营模型
- AI 自动派车正式落地
- OCR 识票、票据查验、影像归档
- 完整 BI 平台级分析

### 3.3 第一阶段可作为 P1 的能力

以下能力允许进入 `P1` 储备，但不进入开工承诺主排期：

- 月度核算清单导出
- 看板增强
- 通知失败重试可视化
- 规则配置后台增强
- 异常识别预警
- Agent 查询助手入口

## 4. 第一期开工冻结业务口径

### 4.1 业务定位冻结

- 新系统是业务主系统和唯一业务主状态源
- 飞书是统一入口、审批协同入口和消息通知入口
- OA 在第一期中不是默认主流程系统
- Agent 不是第一期交付物，不参与主流程核心判断

### 4.2 申请类型冻结

第一期仅支持两类申请：

- `OFFICIAL`：公务用车
- `PRIVATE`：私车公用

不支持更多细分申请类型，例如“接送站”“短驳”“跨城专项”等独立流程模型。

### 4.3 角色冻结

第一期角色冻结如下：

- 普通员工
- 一级审批人
- 二级审批人
- 行政经理
- 行政司机
- 系统管理员

不单独新增“财务复核人”“车队长”“超级审批人”等额外业务角色。

### 4.4 审批链冻结

- 一级审批人默认取申请人所在部门负责人
- 二级审批人默认取行政经理
- 审批人来源默认取飞书组织架构
- 第一阶段默认两级审批
- 支持审批撤回、移交、加签、会签，但这些动作记录在审批记录中，不改变主状态设计方法

### 4.5 组织模型冻结

- 所有公务车辆统一归属总裁办管理
- 所有行政司机统一归属总裁办管理
- 第一阶段不支持跨车队、跨管理主体的资源归属模型

## 5. 冻结的关键范围决策

### 5.1 业务范围决策表

| 决策项 | 冻结结论 | 说明 |
| --- | --- | --- |
| 审批承载位置 | 飞书 | 审批动作在飞书，主业务状态在新系统 |
| 申请填写位置 | 新系统 | 飞书只做入口和审批，不做主业务表单 |
| 派车位置 | 新系统 | 不在飞书审批表单中直接派车 |
| 私车公用核验位置 | 新系统 | 行政经理在系统内核验，不在飞书处理 |
| 主业务状态来源 | 新系统 | 飞书审批状态仅作为协同事件 |
| 数据导出 | 非 P0 | 进入 P1 储备 |
| Agent 正式交付 | 非 P0 | 仅预留接口和扩展位 |

### 5.2 状态模型冻结

第一期主申请单状态冻结为：

- `DRAFT`
- `PENDING_L1`
- `PENDING_L2`
- `REJECTED`
- `PENDING_DISPATCH`
- `RESERVED`
- `IN_PROGRESS`
- `PENDING_CONFIRM`
- `COMPLETED`
- `CANCELLED`
- `APPROVAL_EXCEPTION`

说明：

- `APPROVAL_EXCEPTION` 用于飞书审批创建失败、回调长时间异常或人工补偿前的中间态
- 如实现时确实不希望暴露在前台页面，可只在后台和日志中可见，但数据库与服务层应保留该状态

### 5.3 取消与变更冻结

- 申请人在审批前和审批中可取消
- 审批通过待派车状态允许取消
- 已预占但未实际出车状态允许取消，但必须释放资源
- 执行中不允许普通员工直接取消
- 已完成、已取消状态不可再次取消
- 变更采用“生成变更单并重新审批”的模式，不直接覆盖原单

## 6. 未决事项冻结建议

以下事项在现有文档中属于“曾提出但未完全锁死”的内容。为保证开工，先冻结默认口径如下。

### 6.1 司机是否允许手工改派

冻结建议：

- 支持行政经理在“未开始执行”前手工改派车辆和司机
- 改派必须重新做冲突校验
- 改派后保留原派车记录与新派车记录
- 执行中不支持直接改派

默认推进结论：`纳入 P0`

原因：

- 这是实际派车场景中的高频操作
- 若完全不支持，行政经理只能取消后重建，体验和数据都较差

### 6.2 是否需要上传费用票据原件

冻结建议：

- 第一阶段支持上传电子附件
- 对存在报销费用的记录，支持上传一张或多张附件
- 第一阶段不做票据原件真实性校验
- 第一阶段不做 OCR、发票查验、财务影像归档集成

默认推进结论：`纳入 P0，能力做轻`

### 6.3 是否保留 OA 流程编号映射

冻结建议：

- 第一阶段默认不要求每一张新系统申请都同步生成 OA 流程编号
- 若个别制度要求必须留痕在 OA，则作为例外场景单独设计
- 系统保留 `external_ref_no` 或等价扩展字段，供未来接入 OA 时使用

默认推进结论：`不纳入 P0 主链路`

### 6.4 是否需要导出月度核算清单

冻结建议：

- 月度核算清单导出进入 `P1`
- 第一阶段先保证核算结果可查、规则可追溯、列表可筛选

默认推进结论：`不纳入 P0`

### 6.5 私车公用导航里程如何获取

冻结建议：

- 第一阶段由行政经理手工录入导航核验里程
- 页面保留“导航来源说明/截图备注”字段
- 第一阶段不集成地图 API 自动拉取路线里程

默认推进结论：`纳入 P0，采用人工核验`

### 6.6 是否需要审批人查看完整费用明细

冻结建议：

- 审批阶段仅展示申请必要字段
- 不在飞书审批表单中展示消耗、补助、核算等执行后数据
- 审批人如需更多信息，通过详情链接跳转系统查看申请详情

默认推进结论：`按轻审批模型执行`

## 7. 可开工的前置确认清单

以下事项建议在正式排期会前由业务、产品、技术共同确认并签字：

| 事项 | 当前基线 | 是否阻塞开工 |
| --- | --- | --- |
| 第一阶段是否仅做两类申请 | 是 | 否 |
| 是否默认两级审批 | 是 | 否 |
| 飞书是否作为审批主入口 | 是 | 否 |
| OA 是否退出主流程 | 是 | 否 |
| 手工改派是否纳入 P0 | 是 | 否 |
| 导出是否移到 P1 | 是 | 否 |
| 地图里程是否先人工核验 | 是 | 否 |
| 财务系统是否一期对接 | 否 | 否 |

## 8. 仍需管理的非阻塞未决事项

以下事项不阻塞脚手架、数据库、主流程和大部分页面开发，但建议在对应节点前补齐：

| 事项 | 建议确认节点 | 默认口径 |
| --- | --- | --- |
| 电子附件数量与大小限制 | 页面联调前 | 单次最多 9 个附件 |
| 司机补助取整规则 | 核算开发前 | 先保留两位小数 |
| 私车公用偏差阈值 | 私车核验页面开发前 | 默认 10 公里高亮提醒 |
| 看板统计口径 | 看板开发前 | 按申请单维度统计 |
| 消息模板文案 | 联调前 | 技术先按占位模板开发 |

## 9. 变更控制要求

第一期开工后，以下变更应视为高影响变更，必须走评审：

- 审批层级从 2 级改为可配置多级
- 引入 OA 并行主流程
- 补助规则计算模型发生结构变化
- 私车公用从“单段行程”改为“多段行程明细”
- 车辆归属和司机归属模型改为多车队
- 审批动作从飞书切回新系统自建审批引擎

## 10. 当前冻结结论

截至当前版本，项目已经具备正式开工条件，默认按以下口径推进：

- 第一阶段以公务用车和私车公用两条主链路为核心
- 飞书承担入口、审批、通知；新系统承担主业务状态和业务执行
- OA 不进入第一期主链路
- 手工改派纳入 P0
- 附件上传纳入 P0，但不做财务级票据能力
- 月度导出不纳入 P0
- 导航里程先走人工核验
