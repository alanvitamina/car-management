# 公务用车管理系统：接口设计文档 V1

> ⚠️ 本文档为 V1 设计阶段产物，合并自《接口清单V1》和《接口详细字段定义V1》。实际实现以 CLAUDE.md 中的 API 路由表为准。

## 1. 设计原则

- 业务接口与集成接口分层
- 读写接口按资源模块清晰划分
- 关键动作接口要有明确状态校验
- 审批回调、通知发送等异步接口单独归类
- 第一版优先满足 Web 和飞书嵌入场景

## 2. 统一约定

### 2.1 响应结构

```json
{ "code": "0", "message": "success", "data": {} }
```

`code = 0` 成功，非 0 失败。

### 2.2 分页结构

```json
{ "code": "0", "message": "success", "data": { "list": [], "page_no": 1, "page_size": 20, "total": 100 } }
```

### 2.3 时间格式

- 日期时间：`yyyy-MM-dd HH:mm:ss`
- 日期：`yyyy-MM-dd`

### 2.4 状态值

| 分类 | 状态值 |
|------|--------|
| 申请类型 | `OFFICIAL`, `PRIVATE` |
| 申请状态 | `DRAFT`, `PENDING_L1`, `PENDING_L2`, `REJECTED`, `PENDING_DISPATCH`, `RESERVED`, `IN_PROGRESS`, `PENDING_CONFIRM`, `COMPLETED`, `CANCELLED`, `APPROVAL_EXCEPTION` |
| 审批状态 | `PENDING`, `APPROVED`, `REJECTED`, `WITHDRAWN` |
| 车辆状态 | `IDLE`, `RESERVED`, `IN_USE`, `MAINTENANCE`, `DISABLED` |
| 司机状态 | `AVAILABLE`, `RESERVED`, `IN_USE`, `REST`, `LEAVE`, `DISABLED` |

## 4. 核心接口详细定义

### 4.1 获取当前用户信息

- `GET /api/auth/me`

**响应字段**：user_id, user_name, feishu_user_id, department_id, department_name, role_tags, permissions

### 4.2 获取审批人信息

- `GET /api/departments/approvers?user_id={id}`

**响应字段**：applicant_user_id, first_approver_user_id, first_approver_name, second_approver_user_id, second_approver_name, rule_source

业务约束：一级审批人默认取部门负责人，二级审批人默认取行政经理。组织数据缺失时返回 `APP_002`。

### 4.3 查询可用车辆

- `GET /api/vehicles/available?start_time={}&end_time={}`

**响应字段**：available_list（vehicle_id, plate_no, vehicle_model, vehicle_status）, conflict_list（含 conflict_reason）

### 4.4 提交申请

- `POST /api/applications`

**请求字段**：use_car_type(必填), start_time(必填), end_time(必填), purpose(必填), destination(必填), remark(选填)

**响应字段**：application_id, application_no, application_status, first_approver_name, second_approver_name, approval_instance_id

业务约束：start_time 必须早于 end_time，提交后自动发起飞书审批，飞书审批创建失败时进入审批异常状态。可能错误码：`APP_002`, `APP_003`, `INTEGRATION_001`。

### 4.5 查询申请列表

- `GET /api/applications`

**请求参数**：page_no, page_size, applicant_user_id, use_car_type, application_status, start_date, end_date

**列表项字段**：application_id, application_no, use_car_type, start_time, end_time, destination, application_status, applicant_user_name

### 4.6 获取申请详情

- `GET /api/applications/{id}`

**响应分为 5 个区块**：
- `basic_info`：申请基本信息
- `approval_info`：审批人、审批状态、审批记录列表
- `dispatch_info`：车辆ID/车牌号、司机ID/姓名、派车状态
- `consumption_info`：记录人、起止地点、里程、费用、确认状态
- `subsidy_info`：核算列表、补助类型、金额、规则版本

### 4.7 取消申请

- `POST /api/applications/{id}/cancel`

**请求字段**：cancel_reason(必填)

业务约束：仅 DRAFT/PENDING_L1/PENDING_L2/PENDING_DISPATCH/RESERVED 状态可取消。已预占需释放资源，审批中需撤回飞书审批实例。

### 4.8 发起变更申请

- `POST /api/applications/{id}/change`

### 4.9 复制原单重新发起

- `POST /api/applications/{id}/recreate`

### 4.10 飞书审批回调

- `POST /api/approvals/feishu/callback`

必须幂等处理。原始报文先落库，再归一化为内部审批事件对象。

**内部事件对象字段**：approval_instance_id, application_id, application_no, approval_node_type, approver_user_id, action_type, action_comment, acted_at, raw_payload

**业务反查优先级**：application_id → application_no → approval_instance_id

### 4.11 撤销/移交/加签

- `POST /api/approvals/{application_id}/withdraw`
- `POST /api/approvals/{application_id}/transfer`
- `POST /api/approvals/{application_id}/add-sign`

### 4.12 查询待派车列表

- `GET /api/dispatch/pending`

**列表项字段**：application_id, application_no, applicant_user_name, department_name, start_time, end_time, destination, available_vehicle_count, available_driver_count

### 4.13 冲突校验

- `POST /api/dispatch/check-conflict`

**请求字段**：application_id, vehicle_id, driver_id

**响应字段**：can_assign, vehicle_conflict, driver_conflict, conflict_reasons

### 4.14 确认派车

- `POST /api/dispatch/assign`

**请求字段**：application_id, vehicle_id, driver_id, remark(选填)

**响应字段**：dispatch_id, application_status, vehicle_status, driver_status

业务约束：仅 PENDING_DISPATCH 状态允许，派车时必须二次校验，成功后进入 RESERVED。

### 4.15 释放资源

- `POST /api/dispatch/{application_id}/release`

### 4.16 消耗记录照片上传

- `POST /api/consumptions/photos/upload`（multipart/form-data）

**请求字段**：file(必填), photo_type(必填, `DEPARTURE`/`RETURN`)

**响应字段**：photo_url, exif_time, time_source

文件格式支持 JPG/PNG/HEIC，大小限制 20MB。EXIF 缺失时返回 null。

### 4.17 公务用车提交消耗记录

- `POST /api/consumptions/official`

**请求字段**：application_id(必填), start_mileage(必填), end_mileage(必填), departure_photo_url(必填), return_photo_url(必填), actual_departure_at(选填), actual_return_at(选填), fuel_fee, toll_fee, parking_fee, other_fee, remark

自动计算：actual_mileage = end_mileage - start_mileage；后端从照片提取 EXIF 时间。

业务约束：照片必须上传，end_mileage ≥ start_mileage，提交后进入"待确认"。

### 4.18 私车公用提交里程与费用

- `POST /api/consumptions/private`

**请求字段**：application_id(必填), start_place(必填), end_place(必填), start_mileage(必填), end_mileage(必填), departure_photo_url(必填), return_photo_url(必填), toll_fee, parking_fee, other_fee, remark

### 4.19 消耗确认/驳回

- `POST /api/consumptions/{application_id}/confirm`
- `POST /api/consumptions/{application_id}/reject`（请求字段：reject_reason）

确认后数据锁定，自动触发司机补助核算，恢复车辆和司机状态。

### 4.20 私车公用核验

- `POST /api/consumptions/{application_id}/verify-private-mileage`

**请求字段**：navigation_mileage(必填), settlement_mileage(必填), verification_comment(选填)

**响应字段**：application_id, actual_mileage, navigation_mileage, settlement_mileage, subsidy_amount

仅行政经理可调用，核验后可查看里程照片原图。核验后自动触发私车公用补助核算。

### 4.21 核算管理

- `GET /api/subsidies` — 核算列表
- `GET /api/subsidies/{application_id}` — 核算详情（settlement_id, settlement_type, target_user_name, base_value, multiplier, amount, rule_version, generated_at）
- `POST /api/subsidies/calculate` — 触发核算
- `POST /api/subsidies/export` — 导出

### 4.22 补助规则 CRUD

- `GET /api/subsidy-rules`
- `POST /api/subsidy-rules`
- `PUT /api/subsidy-rules/{id}`

### 4.23 工作台摘要

- `GET /api/dashboard/summary`

**响应字段**：pending_approval_count, pending_dispatch_count, pending_confirm_count, pending_verify_count, my_application_count, idle_vehicle_count, reserved_vehicle_count, in_use_vehicle_count

### 4.24 看板统计

- `GET /api/dashboard/vehicle-status`
- `GET /api/dashboard/application-trend`
- `GET /api/dashboard/vehicle-ranking`
- `GET /api/dashboard/department-usage`
- `GET /api/dashboard/subsidy-summary`

### 4.25 其他接口

**车辆管理**：`GET/POST /api/vehicles`, `GET/PUT /api/vehicles/{id}`, `PUT /api/vehicles/{id}/status`

**司机管理**：`GET/POST /api/drivers`, `GET/PUT /api/drivers/{id}`, `PUT /api/drivers/{id}/status`

**部门与组织**：`GET /api/departments/tree`, `POST /api/departments/sync-feishu`

**系统配置**：`GET/POST /api/system-configs`, `GET/POST /api/system-configs/notification-templates`

**集成日志**：`GET /api/logs/sync`, `GET /api/logs/integration`, `GET /api/logs/notifications`

## 5. 关键调用链

### 提交申请
1. 前端 `GET /api/departments/approvers` 获取审批人
2. 前端 `POST /api/applications` 提交申请
3. 后端创建 `car_application` → 调用飞书创建审批实例 → 写入 `car_approval_record`

### 审批回调
1. 飞书 → `POST /api/approvals/feishu/callback`
2. 后端解析 → 更新 `car_approval_record` → 更新 `car_application` 主状态 → 发送通知

### 派车
1. 前端 `POST /api/dispatch/check-conflict` → `POST /api/dispatch/assign`
2. 后端写入 `car_dispatch_record` → 更新车辆/司机状态 → 通知

### 私车核验
1. 用车人 `POST /api/consumptions/private`
2. 行政经理 `POST /api/consumptions/{id}/verify-private-mileage`
3. 后端触发 `POST /api/subsidies/calculate` → 写入 `car_subsidy_settlement`

## 6. 错误码

| 错误码 | 说明 |
|--------|------|
| `APP_001` | 申请状态不允许当前操作 |
| `APP_002` | 审批人未找到 |
| `APP_003` | 申请时间非法 |
| `DISPATCH_001` | 车辆已被占用 |
| `DISPATCH_002` | 司机已被占用 |
| `CONSUME_001` | 消耗记录状态不允许操作 |
| `SUBSIDY_001` | 补助规则未配置 |
| `INTEGRATION_001` | 飞书审批发起失败 |
| `INTEGRATION_002` | 飞书回调解析失败 |

## 7. 联调注意事项

- 审批回调必须做签名校验和幂等控制
- 审批人以后端计算结果为准，前端不自行拼凑
- 可用车辆查询仅作前端参考，派车前必须后端二次校验
- 实际补助必须使用 `settlement_mileage`，非 `actual_mileage`
- 补助计算逻辑封装为独立服务，接口返回携带规则版本和计算明细
- 导航里程与记录里程偏差较大时前端高亮提示
