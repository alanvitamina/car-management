# 公务用车管理系统：飞书与 OA 集成设计说明 V2

## 1. 文档说明

本文档用于把现有飞书审批字段映射材料升级为可联调、可开发、可验收的集成设计说明，覆盖飞书登录、组织同步、审批创建、审批回调、消息通知、OA 保留策略和联调要求。

本文件是第一期外部系统集成的主说明文档。

## 2. 集成设计结论

第一期采用以下总体集成模式：

- 员工从飞书进入系统
- 新系统承载业务表单、业务状态和业务执行
- 飞书承载审批动作和通知触达
- 飞书审批结果通过回调写回新系统
- OA 默认不进入第一期主流程，只保留兼容空间

一句话边界：

- 飞书负责“进来、审批、通知”
- 新系统负责“建单、派车、核验、核算、留痕”
- OA 负责“暂不迁移场景或历史保留”

## 3. 集成范围

### 3.1 第一期开启的飞书能力

- 飞书应用入口
- 飞书登录或免登识别
- 飞书组织架构同步
- 飞书审批模板
- 飞书审批实例创建
- 飞书审批回调
- 飞书消息通知

### 3.2 第一阶段不做的飞书能力

- 在飞书表单中直接编辑新系统主业务字段
- 在飞书中直接派车
- 在飞书中直接核验私车公用里程
- 飞书内完整替代新系统详情页

### 3.3 第一阶段对 OA 的定位

- OA 不是默认审批主流程系统
- OA 仅作为可选保留系统存在
- 若个别制度要求必须留痕在 OA，后续通过扩展字段和例外流程兼容，不反向改变第一期主架构

## 4. 总体架构与时序

### 4.1 总体架构

```text
飞书用户
  |
  v
飞书应用入口
  |
  v
新系统 Web 前端
  |
  v
新系统后端
  |          \
  |           \__ Redis（幂等、锁、短期缓存）
  |
  +-- MySQL（业务主数据、审批记录、集成日志）
  |
  +-- 飞书开放平台
  |     |- 登录识别
  |     |- 组织架构
  |     |- 审批实例
  |     |- 回调事件
  |     \- 消息通知
  |
  \-- OA（第一期默认不在主链路）
```

### 4.2 提交申请到审批完成时序

1. 用户从飞书进入新系统并识别身份
2. 新系统创建业务申请单
3. 新系统计算审批人并选择审批模板
4. 新系统调用飞书创建审批实例
5. 飞书审批人完成一级、二级审批
6. 飞书回调审批事件到新系统
7. 新系统归一化处理回调事件并推进主状态
8. 公务用车进入待派车，私车公用进入待核验/待执行

## 5. 身份与组织集成设计

### 5.1 登录识别原则

- 系统以飞书身份作为统一入口身份源
- 新系统必须维护“飞书用户 ID 与系统用户 ID”的映射
- 前端不得自行拼凑用户身份，所有角色和权限以后端返回为准

### 5.2 用户唯一键建议

建议优先使用以下标识进行映射：

- 飞书用户唯一 ID
- 系统用户主键
- 员工工号或等价企业人员编号

说明：

- 若飞书用户 ID 可能因租户或环境差异变化，应同时保留稳定的企业人员编号作为辅助匹配键

### 5.3 组织同步策略

- 组织架构数据默认来自飞书
- 至少同步部门、用户、部门负责人、启停状态
- 同步方式第一阶段可采用定时拉取
- 同步失败必须记录集成日志并可人工重试

### 5.4 审批人带出依赖

审批人自动带出依赖以下组织数据完整：

- 用户所在部门
- 部门负责人
- 行政经理配置

缺一不可。若缺失，应阻断提交并提示管理员处理。

## 6. 飞书审批模板设计

### 6.1 模板数量

第一阶段建议建立 2 个审批模板：

1. `公务用车申请审批`
2. `私车公用申请审批`

### 6.2 模板选型原则

- 按申请类型分模板
- 模板只承载审批判断必要字段
- 不把执行数据和核算数据放入审批模板

### 6.3 审批模板字段

两类模板第一阶段建议统一展示以下字段：

- 申请单号
- 申请人
- 申请部门
- 用车类型
- 开始时间
- 结束时间
- 用车事由
- 目的地
- 备注
- 业务详情链接

### 6.4 模板不承载的字段

以下字段不进入飞书审批主表单：

- 可用车辆列表
- 分配车辆
- 分配司机
- 导航核验里程
- 核算里程
- 私车公用补助金额
- 司机补助明细

## 7. 新系统到飞书的字段映射

### 7.1 公共字段映射

| 新系统字段 | 飞书字段建议名 | 是否必传 | 说明 |
| --- | --- | --- | --- |
| `application_id` | `biz_id` 或隐藏业务 ID | 是 | 业务主键回查 |
| `application_no` | `申请单号` | 是 | 人工识别与兜底回查 |
| `applicant_user_name` | `申请人` | 是 | 审批展示 |
| `department_name` | `申请部门` | 是 | 审批展示 |
| `use_car_type` | `用车类型` | 是 | 公务用车/私车公用 |
| `start_time` | `开始时间` | 是 | 审批判断 |
| `end_time` | `结束时间` | 是 | 审批判断 |
| `purpose` | `用车事由` | 是 | 审批判断 |
| `destination` | `目的地` | 是 | 审批判断 |
| `remark` | `备注` | 否 | 补充说明 |
| `application_detail_url` | `详情链接` | 是 | 审批人查看系统详情 |

### 7.2 审批人映射

| 新系统字段 | 飞书节点配置 | 说明 |

## 8. 飞书回调设计

### 8.1 回调处理总原则

- 原始回调报文必须先落库
- 业务代码不直接依赖飞书原始结构
- 所有回调统一转换为内部审批事件对象
- 回调处理必须幂等

### 8.2 内部统一审批事件对象

建议统一抽象为以下字段：

| 字段 | 说明 |
| --- | --- |
| `approval_instance_id` | 飞书审批实例 ID |
| `application_id` | 申请单 ID |
| `application_no` | 申请单号 |
| `approval_node_type` | 一级、二级、加签、会签 |
| `approver_user_id` | 系统审批人 ID |
| `approver_feishu_user_id` | 飞书审批人 ID |
| `action_type` | 通过、驳回、撤回、移交、加签、会签 |
| `action_result` | 成功、失败、待定 |
| `action_comment` | 审批意见 |
| `acted_at` | 动作时间 |
| `raw_payload` | 原始报文 |

### 8.3 回调业务反查优先级

建议按以下顺序反查业务单：

1. `application_id`
2. `application_no`
3. `approval_instance_id`

### 8.4 回调签名校验

- 回调接口必须校验来源合法性
- 验签失败时不得推进业务状态
- 验签失败事件必须进入集成日志

### 8.5 幂等规则

- 同一回调事件重复推送，只允许首次生效
- 幂等键建议采用“平台事件唯一 ID”优先
- 若平台未提供稳定唯一事件 ID，则采用“审批实例 ID + 节点类型 + 审批人 + 动作时间”组合

### 8.6 回调失败补偿

回调处理失败时：

- 原始数据保留
- 写入集成日志
- 标记为待补偿
- 支持人工重放或后台自动重试

## 9. 状态映射

### 9.1 飞书审批状态到系统审批状态

| 飞书动作/状态 | 系统审批记录状态 | 主申请单状态影响 |
| --- | --- | --- |
| 待审批 | `PENDING` | 不变 |
| 一级通过 | `APPROVED` | `PENDING_L1 -> PENDING_L2` |
| 一级驳回 | `REJECTED` | `PENDING_L1 -> REJECTED` |
| 二级通过且为公务用车 | `APPROVED` | `PENDING_L2 -> PENDING_DISPATCH` |
| 二级通过且为私车公用 | `APPROVED` | `PENDING_L2 -> PENDING_CONFIRM` |
| 二级驳回 | `REJECTED` | `PENDING_L2 -> REJECTED` |
| 申请人撤回 | `WITHDRAWN` | `PENDING_L1/PENDING_L2 -> CANCELLED` |
| 移交 | `TRANSFERRED` | 主状态不变 |
| 加签中 | `ADD_SIGNING` | 主状态不变 |
| 会签中 | `COUNTER_SIGNING` | 主状态不变 |

### 9.2 系统申请状态到飞书展示关系

- 飞书审批只展示审批过程状态
- 新系统详情页展示完整业务状态
- 不要求把所有主业务状态同步回飞书审批单据

## 10. 通知设计

### 10.1 飞书通知触发点

第一阶段建议至少配置以下通知：

- 申请提交成功后通知一级审批人
- 一级审批通过后通知二级审批人
- 审批驳回/通过后通知申请人
- 派车成功后通知申请人和司机
- 消耗待确认时通知用车人
- 私车公用待核验时通知行政经理

### 10.2 通知失败策略

- 失败时记录集成日志
- 支持重试
- 第一阶段允许后台人工触发重发

## 11. OA 集成策略

### 11.1 第一阶段结论

- 第一阶段不把 OA 作为默认审批承载平台
- 不要求每一笔新申请都生成 OA 流程编号
- 不要求 OA 驱动新系统主状态变化

### 11.2 保留扩展点

系统设计时建议保留以下扩展能力：

- `external_ref_no` 外部流程编号
- `external_system_code` 外部系统编码
- `sync_status` 同步状态
- `sync_payload` 外部同步摘要

### 11.3 适合保留在 OA 的例外场景

- 历史流程归档要求必须留痕在 OA
- 制度要求必须走 OA 的高敏感审批
- 暂时无法迁移出 OA 的存量行政流程

说明：

- 这些是例外兼容，不应推翻第一期“新系统主业务 + 飞书审批”的主架构

### 12.1 主要业务表

飞书/OA 集成主要涉及以下表：

- `car_application`
- `car_approval_record`
- `car_application_operation_log`
- `sys_integration_log`
- `sys_user`
- `sys_department`

### 12.2 建议新增或明确的关键字段

| 表 | 字段建议 | 用途 |
| --- | --- | --- |
| `car_application` | `current_approval_instance_id` | 当前飞书审批实例 ID |
| `car_application` | `approval_exception_flag` | 审批异常标记 |
| `car_application` | `external_ref_no` | 预留 OA 编号 |
| `car_approval_record` | `callback_payload` | 原始回调报文 |
| `sys_integration_log` | `external_system` | 飞书/OA 标识 |
| `sys_integration_log` | `retry_count` | 重试次数 |

### 13.1 对内业务接口

- `GET /api/auth/me`
- `POST /api/applications`

- `GET /api/applications/{id}`

### 13.2 对外集成接口

- `POST /api/approvals/feishu/create`
- `POST /api/approvals/feishu/callback`
- `POST /api/approvals/{application_id}/withdraw`
- `POST /api/notifications/feishu/send`

说明：

- 实际是否暴露为内部 API 还是应用服务方法，可由开发团队按架构规范处理

## 14. 配置清单

第一阶段建议至少准备以下配置项：

- 飞书应用 `app_id`
- 飞书应用 `app_secret`
- 飞书回调验签配置
- 飞书审批模板 ID
- 飞书消息模板或消息发送配置
- 组织同步定时任务开关
- 行政经理默认角色或用户映射配置
- OA 扩展开关

敏感配置必须通过环境变量或密钥管理方式注入，不得写死在仓库中。

## 15. 联调与验收要求

### 15.1 联调前置条件

- 飞书测试应用已创建
- 审批模板已配置并可发起实例
- 测试用户、审批人、行政经理、司机账号已准备
- 回调地址已联通
- 测试环境可访问飞书回调入口

### 15.2 联调必测场景

- 登录识别正确
- 一级审批通过
- 二级审批通过
- 一级审批驳回
- 二级审批驳回
- 申请人撤回
- 审批移交
- 加签/会签回调留痕
- 重复回调幂等
- 回调失败补偿
- 通知发送失败重试

### 15.3 验收标准

满足以下条件后，飞书集成可视为通过：

- 审批实例创建稳定
- 回调可正确回写
- 主状态与审批记录一致
- 重复回调不重复推进状态
- 集成日志可查可定位

## 16. 当前结论

第一期推荐明确按以下口径实现：

- 飞书只做入口、审批和通知
- 新系统始终是主业务系统和主状态源
- 审批事件统一归一化处理并保留原始报文
- OA 不进入默认主链路，但预留兼容字段

这套设计足以支撑开发团队直接开展登录、审批、回调、通知、组织同步和异常补偿开发。
